# AGENTS.md

## Working style

- Deliver the requested solution first.
- When a task required course-correction, a false start, a reverted approach, repeated failing tests, or a later clearly better approach, run `$solution-retrospective` before ending the task.
- Keep the retrospective short and action-oriented.

## Persistence rules

- Put **repo-specific durable guidance** in this repository's nearest `AGENTS.md`.
- Put **cross-repo reusable workflows** in a **personal skill**, not in this repo skill directory.
- Prefer **updating an existing rule or skill** over creating a new one.
- Create a new skill only when the pattern is reusable, concrete, and likely to save future work.
- Do not create noise: no new skill for one-off mistakes, temporary outages, or facts already enforced by tests/linters.

## Retrospective trigger examples

Run `$solution-retrospective` when one or more are true:

- The first design choice was wrong and had to be replaced.
- The first patch passed partially but violated project conventions.
- Several files were read or changed unnecessarily before the correct path became clear.
- A better verification method was discovered late.
- The same kind of mistake appeared more than once in the task.

## Update rules

When updating this file:

- Add only durable rules.
- Keep additions specific and short.
- Place rules near the scope where they apply.
- Prefer command examples and path hints over vague prose.

- Agent handoff: keep a local chronological troubleshooting log in `tmp/agent-handoff.md`; append findings, fixes, results, and next blockers before ending substantial debug sessions so later agents do not repeat the same steps.
- Agent handoff usage: for follow-up debugging or incident analysis, read the latest relevant `tmp/agent-handoff.md` entries before new triage; use that history first when forming hypotheses about regressions or repeated failures.
- Operator docs sync: when adding or changing `open-*`, `close-*`, `check-*`, or local port targets in [make-tasks/ops.mk](/home/dmytro/workspace/personal_projects/AgenticNativePlatform/make-tasks/ops.mk), update the matching local access sections in [README.md](/home/dmytro/workspace/personal_projects/AgenticNativePlatform/README.md) and [docs/commands.md](/home/dmytro/workspace/personal_projects/AgenticNativePlatform/docs/commands.md) in the same change.
- Flux/Kustomize: prefer directory references with a local `kustomization.yaml` over sibling file references such as `../foo/bar.yaml`; verify with `kubectl kustomize <path>`.
- Flux/Kustomize: shared resources outside a cluster root should be exposed as directory bases with their own `kustomization.yaml`; `configMapGenerator` file sources and direct file resources outside the current root trip Kustomize load restrictions in this repo.
- Flux/Kustomize: mode-specific Flux sync roots under `clusters/*/{external,sops}` must render only the staged Flux `Kustomization` CRs (`platform-infrastructure`, `platform-secrets`, `platform-runtime`, `platform-applications`); do not inline `../infrastructure` or `../apps` there or CRD consumers can dry-run before Helm installs their CRDs.
- Flux/Kustomize: when charts install CRDs that repo manifests consume, render staged Flux `Kustomization` resources so HelmReleases reconcile before dependent custom resources; do not place both in the same Flux apply path.
- Flux staging: `platform-secrets` must not depend on `platform-infrastructure` readiness in this repo; several infrastructure HelmReleases consume secrets during install. Keep `platform-runtime` depending on both `platform-infrastructure` and `platform-secrets`, and keep `platform-applications` depending on `platform-runtime` (plus `platform-secrets` when the root still needs an explicit secret-stage edge).
- Flux HelmRelease + Prometheus CRDs: any HelmRelease staged in `platform-infrastructure` that renders `ServiceMonitor` or `PodMonitor` objects must `dependsOn: kube-prometheus-stack` (or move to a later stage); first-cluster bootstrap in this repo otherwise fails dry-run/install before the monitoring CRDs exist.
- Flux staging: gateway-backed runtime CR consumers (`infrastructure/network/kgateway/resources`, `apps/ai-gateway/agentgateway/resources`, `apps/platform/kmcp/resources`) belong in a dedicated `platform-runtime` stage that depends on both `platform-infrastructure` and `platform-secrets`; keep `RemoteMCPServer` manifests in `platform-applications` and make `platform-applications` depend on `platform-runtime`.
- Flux staging: when `Agent` manifests in `platform-applications` reference `RemoteMCPServer` objects from the same repo, do not rely on plain kustomize `resources:` order; kustomize can still emit `Agent` before `RemoteMCPServer`. Serialize them with child Flux `Kustomization` CRs and `dependsOn` inside the `platform-applications` stage.
- Flux/Kustomize: when a third-party controller chart comes from an upstream GitRepository but this repo still needs local CRs for that API, split it into `apps/.../{core,resources}`; stage `core` in `clusters/*/infrastructure` and the CR consumers in `clusters/*/apps` instead of vendoring the whole chart just to co-locate one manifest.
- Flux HelmRelease: when `chart.spec.chart` points to a repo-local path under `./charts/...`, use `chart.spec.reconcileStrategy: Revision` or Flux will not rebuild the packaged chart on file changes unless `Chart.yaml` version also changes.
- Cluster restart flow: when resuming a staged Flux stack after `cluster-stop`, reconcile `platform-infrastructure` first, then force-reconcile HelmReleases, then wait on `platform-secrets` and `platform-applications`; waiting on higher stages before Helm has restored scaled-down Deployments produces misleading "stuck" reconcile output.
- Pause/resume flow: when `cluster-pause` scales Helm-managed Deployments or StatefulSets to zero, snapshot the replica targets first and restore them before the resume reconcile waits; Helm server-side apply does not reliably take back `spec.replicas` from a prior `kubectl scale`, and HPA-managed Deployments stay `ScalingDisabled` at zero until something raises replicas again.
- Pause-state snapshots: keep `ConfigMap/flux-system/cluster-pause-state` as an exact pre-pause replica snapshot; do not silently coerce zeroed workloads to `1` during normal resume. Reject stale all-zero restores, avoid overwriting a last-known-good snapshot with an already-zeroed pause run, and only use `0 -> 1` recovery as an explicit stale-snapshot fallback so Flux/Helm can converge again.
- Flux bulk resume: for repo-wide HelmRelease fan-out, prefer `reconcile.fluxcd.io/requestedAt` / `forceAt` / `resetAt` annotations plus staged Kustomization waits over a long sequential `flux reconcile helmrelease ...` loop.
- Operator Make targets: for user-facing `kubectl` / `flux` Make targets, prefer explicit `--kubeconfig "$(KUBECONFIG)"` binding over relying only on the exported environment; it makes `make -n` and README snippets match what actually happens.
- Operator Make targets: when a preflight reads Services, Endpoints, or Flux objects, treat `kubectl` command failures separately from empty results; surface API-server-unreachable errors explicitly instead of reporting generic `Missing` or `no ready endpoints`.
- Operator Make targets: when a Make target shells out to repo scripts and users may override inputs on the command line (`make ... VLLM_IMAGE=...`, `ECHO_MCP_IMAGE=...`, `GIT_BRANCH=...`), pass those variables explicitly into the script invocation; values imported from `.env` inside scripts otherwise override the intended command-line render.
- Topology-aware Make/shell paths: when a target or script accepts `TOPOLOGY`, derive `TF_DIR` from that final topology and source `.env` before computing the fallback; explicit `make ... TOPOLOGY=github-codespace` must not silently keep using `local` paths from ambient variables.
- Makefile topology defaults: normalize an explicitly empty command-line `TOPOLOGY=` back to `local` with `override`; GNU Make treats empty CLI assignments as higher precedence than `?=`, which otherwise leaves `TF_DIR` pointing at `terraform/environments/`.
- Plaintext secret bootstrap: `make apply-plaintext-secrets` runs before Flux bootstrap, so `.generated/secrets/<env>/namespaces.yaml` must include every namespace referenced by those secrets, including early `context` and `observability` objects.
- Flux Git bootstrap preflight: when validating `GIT_REPO_URL` against local remotes, compare normalized repo identity (`github.com/org/repo`) rather than exact SSH-vs-HTTPS URL strings; developers often clone with SSH while Flux reads HTTPS.
- GitHub Actions Gitleaks: for repo hygiene scans, use `gitleaks dir <path>` against the checked-out worktree with the repo `.gitleaks.toml`; this repo contains bundled Finnhub reference assets that create noisy findings in history-wide scans and `dir` takes the target path as a positional argument, not `--source`.
- GitHub Actions Go image scans: when Trivy scans the built Finnhub image, keep the workflow `GO_VERSION`, README snippet, and `mcp/finnhub-mcp-server/Dockerfile` builder tag on the same patched Go release; patch drift leaves CI green on `go test` but fails the image scan on Go stdlib CVEs.
- GitHub Actions GHCR publish: avoid `cache-to=gha` on repo image-publish workflows that also push to GHCR; in this repo the Buildx cache export path failed after image export with GHCR blob `403 Forbidden`, so keep the publish step cacheless unless there is a verified working cache backend.
- GitHub Actions GHCR publish: if `docker/login-action` succeeds but Buildx push still fails with GHCR blob `HEAD ... 403 Forbidden`, treat it as a package-path/ACL problem before assuming the Dockerfile is wrong; in this repo the durable fix was to publish the Finnhub image to a fresh owner-level package path instead of the older nested `ghcr.io/<owner>/agenticnativeplatform/...` path.
- Multi-arch Dockerfiles: when Buildx publishes `linux/amd64,linux/arm64`, do not hardcode `GOARCH`; use `TARGETOS` / `TARGETARCH` args or the arm64 leg will emit an amd64 binary and waste minutes under emulation.
- kgateway image pinning: the `kgateway` OCI chart can render a controller image tag with `+gitsha` suffixes such as `2.1.1+...`; pin `spec.values.controller.image.registry/repository/tag` in `infrastructure/network/kgateway/core/release.yaml` to a stable published tag such as `cr.kgateway.dev/kgateway-dev/kgateway:v2.1.1`.
- kgateway BackendConfigPolicy: if repo runtime manifests use `gateway.kgateway.dev/v1alpha1` `BackendConfigPolicy`, install both `kgateway-crds` and `kgateway` charts and keep the kgateway source on a line that serves that CRD (`v2.1.x+` in this repo).
- kgateway runtime triage: before debugging missing kagent agents or empty Inventory after bootstrap, run `kubectl --kubeconfig .kube/generated/current.yaml apply --dry-run=server -k clusters/local-dev/runtime`; an invalid runtime CR such as `infrastructure/network/kgateway/resources/agentgateway-backend-policy.yaml` keeps `platform-runtime` `NotReady` and blocks `platform-applications`.
- Port-forward cleanup in Make recipes: do not use broad `pgrep -f` patterns that also appear in the current shell command line; match only real `kubectl` processes via `ps ... | awk '$$2=="kubectl"'` before killing stale managed forwards.
- Flux local chart packaging: when repo-local charts use `.Chart.Version` in labels or default image tags, sanitize `+` for labels and pin explicit image tags if the upstream registry does not publish `+gitsha` tags; do not solve this with a blanket `global.tag` override when vendored subcharts publish different image tags.
- Flux local chart packaging: for `charts/vendor/kagent`, pin `controller.image.tag`, `controller.agentImage.tag`, `controller.skillsInitImage.tag`, and `ui.image.tag` explicitly in the HelmRelease values. The top-level `tag` alone is not sufficient to prevent invalid `+gitsha` image references during Flux upgrades.
- Kagent grafana-mcp: `grafana.secretRef` must reference `Secret/kagent-grafana-mcp` in the `kagent` namespace containing `GRAFANA_SERVICE_ACCOUNT_TOKEN` or `GRAFANA_API_KEY`; do not point it at `observability/observability-grafana-admin`, and use the Grafana root URL, not `/api`, for `grafana.url`.
- Kagent grafana-mcp bootstrap: on fresh clusters, treat `Secret/kagent-grafana-mcp` as a runtime-generated secret, not a static Git secret; bring Grafana up first, mint the token from the Grafana API, apply the secret, then restart/reconcile kagent.
- AgentGateway + LiteLLM rate limits: attach `AgentgatewayPolicy.spec.traffic.rateLimit.local` to the `/v1` `HTTPRoute` or Gateway, not the `AgentgatewayBackend`; keep backend timeouts in a separate backend-targeted policy. Mirror the same numeric cap in `values/common/litellm*.yaml` with per-model `rpm` / `tpm`; AgentGateway local limits are per proxy replica, so exact cluster-wide caps assume a single `agentgateway-proxy` pod.
- AgentGateway + service-backed web tools: when exposing an MCP sidecar's human web UI, use a separate `HTTPRoute` directly to the backing `Service` with a dedicated path prefix and `URLRewrite`; do not broaden the MCP `AgentgatewayBackend` route. Add a matching `ReferenceGrant` in the target namespace.
- AgentGateway Admin UI: `http://localhost:<port>/ui/` is the proxy pod admin port `15000`, not the gateway listener `8080`; use a separate `kubectl port-forward deployment/agentgateway-proxy <local>:15000` target instead of expecting the repo's `open-agentgateway` service forward on `8080` to serve `/ui/`.
- Kagent observability-agent: when `tools.grafana-mcp.enabled=false`, also set `observability-agent.grafanaMCP.enabled=false` or the rendered `Agent/observability-agent` keeps a missing `RemoteMCPServer` reference and stays `Accepted=False`.
- LiteLLM image pinning: keep `charts/litellm-proxy` on a concrete LiteLLM image tag; do not use floating tags such as `main-stable` for operator-facing `/v1` APIs because upstream auth/model behavior can change independently of this repo's stateless config.
- Flux local chart packaging: the vendored `charts/vendor/kagent` chart renders its own `default-model-config`, so provider secret and base URL overrides must be set on the main `kagent` HelmRelease, not only on `kagent-crds` or separate `kagent-resources` manifests.
- AgentGateway: current Kubernetes `AgentgatewayBackend` CRDs use `spec.ai` with provider configuration, not legacy `spec.llm`; keep both active and duplicate repo manifests aligned with that schema.
- AgentGateway: current policy CRD is `AgentgatewayPolicy`, not `BackendTrafficPolicy`; backend request timeout belongs under `spec.backend.http.requestTimeout`.
- AgentGateway/Gateway API: when an `HTTPRoute` references `AgentgatewayBackend` from another namespace, set `backendRefs[].namespace` explicitly or the controller resolves it in the route namespace and returns `BackendNotFound` / `no valid backends`.
- AgentGateway/LiteLLM: if routes are resolved but AgentGateway still returns `upstream call failed: Connect: Connection refused`, inspect `ztunnel` and the target pod annotations; for local `ai-gateway` workloads under ambient, opt the backend pod out with `istio.io/dataplane-mode: none` before assuming the app bind address is wrong.
- Declarative kagent Agent manifests: under `spec.declarative.a2aConfig.skills[].examples`, quote any example string containing `:` followed by a space or Flux dry-run can decode it as a map instead of a string.
- Kagent restart behavior: the vendored controller chart's default startup probe is too short for post-stop/start recovery on this repo; keep the controller startup probe configurable and allow at least a few minutes before kubelet restarts it.
- Kagent controller probes: keep startup/readiness on the app server endpoint `:8083/health`; the controller also logs `ProbeAddr :8082`, but probing `:8082/health` returned `404` in this repo's live rollout.
- Kagent + ambient: if `HelmRelease/kagent` stays `InProgress` while the controller pod times out talking to the API server, inspect `ztunnel` and opt the controller pod out with `istio.io/dataplane-mode: none`; the live config may look fine even when ambient blocks startup.
- Kagent MCP identity: do not give an `MCPServer` and a `RemoteMCPServer` the same `namespace/name`; in this repo that caused agentregistry ownership/source collisions on `kagent/finnhub-mcp-server`. Keep the backing KMCP workload on a distinct internal name such as `*-backend` and reserve the public `RemoteMCPServer` name for the AgentGateway-exposed endpoint.
- RemoteMCPServer security: repo-managed `RemoteMCPServer` aliases for agents and inventory must use the AgentGateway URL, not direct backing Service URLs; keep any direct Service exposure limited to separate operator/web routes when explicitly needed.
- RemoteMCPServer integration: a plain REST endpoint under `/mcp` is not enough; use a real MCP server implementation or official MCP reference package, otherwise clients will hit `422` / `405` protocol mismatches.
- RemoteMCPServer + AgentGateway: AgentGateway can rewrite the public MCP session ID between `initialize` and the hanging `GET` stream. Custom Streamable HTTP wrappers must not gate the first-byte SSE preamble on re-seeing the same session ID from the initialize response.
- RemoteMCPServer + AgentGateway: when wrapping a streamable HTTP handler with logging or status middleware, preserve `http.Flusher` on the outer `ResponseWriter`; stripping it can stall `initialize`/SSE flushes so MCPG still looks healthy while kagent and inventory fail through AgentGateway.
- Stop/start flow: if `cluster-stop` scales Helm-managed workloads to zero, `cluster-start` must force-reconcile the affected HelmReleases; resume plus top-level Kustomization reconcile is not enough to restore objects like `Deployment/istiod` from `0/0`.
- Stop/start flow: do not scale `metallb-system` to zero; MetalLB's controller backs the validating webhook for `IPAddressPool`, and zero endpoints blocks `platform-applications` dry-run on the next reconcile.
- Command semantics: use `cluster-pause` / `cluster-resume` for reversible workload suspension, `cluster-remove` for k3s-only removal, and `environment-destroy` for cluster plus Terraform/OpenTofu cleanup; keep `cluster-stop` / `cluster-start` only as compatibility aliases.
- Local OCI cache: `bootstrap-hosts` installs a host-level pull-through cache for `TOPOLOGY=local`; `cluster-remove` keeps it warm for the next rebuild, while `environment-destroy` removes it unless `KEEP_LOCAL_OCI_CACHE=true`.
- Local k3s network repair: `make repair-local-k3s-network TOPOLOGY=local` must restore the GitOps-managed cluster state after reinstalling k3s; do not leave a bare runtime-only recovery path there. When `LOCAL_OCI_CACHE_ENABLED=true`, refresh the host OCI cache service before reinstall so containerd mirror pulls keep using it.
- K3s image handling: local Docker builds are not visible to the cluster until imported into k3s/containerd; for sample images like `echo-mcp`, use tarball import targets and keep the manifest `image:` tag identical to the imported tag.
- K3s image handling: `/var/lib/rancher/k3s/agent/images/` may be absent on fresh nodes; import targets should `mkdir -p` it first instead of assuming it already exists.
- K3s image handling: copying a tarball into `/var/lib/rancher/k3s/agent/images/` is not enough for fast rollouts of a new tag; run `k3s ctr images import <tar>` in the target so kubelet can start the new image immediately instead of attempting a registry pull.
- Local k3s networking: if the workstation switches between Wi-Fi and Ethernet, compare `hostname -I` with `kubectl get node -o wide` before debugging controllers; a stale node `InternalIP` breaks in-cluster API traffic. Prefer `make repair-local-k3s-network TOPOLOGY=local` over app-level pod churn when those IPs differ.
- K3s local disk pressure: in [ansible/templates/k3s-server-config.yaml.j2](/home/dmytro/workspace/personal_projects/AgenticNativePlatform/ansible/templates/k3s-server-config.yaml.j2), prefer small absolute kubelet eviction thresholds such as `imagefs.available<2Gi,nodefs.available<2Gi>` plus explicit image GC thresholds over percentage-based disk eviction on laptop-sized nodes; the earlier `5%` policy in this repo caused `DiskPressure=True` while tens of GiB were still free.
- Ansible privilege escalation: for local ad-hoc `ansible ... -b` commands in this repo, plain `make ...` is preferred; let the local sudo prompt appear if needed, and do not assume `ANSIBLE_BECOME_FLAGS=-K` is required.
- Sample workload image overrides: inject operator-specific images such as `ECHO_MCP_IMAGE` through cluster-scoped wrappers or values sources, not by editing shared component manifests with user-specific tags.
- Sample workload images: the local `mcp/echo-server` image must run with a numeric non-root UID/GID (for example `USER 1000:1000`); `runAsNonRoot` will reject root or named-only users during local kmcp rollouts.
- KMCP packaging: for off-the-shelf MCP packages that KMCP can launch directly (for example npm packages such as `@modelcontextprotocol/server-everything`), prefer `transportType: stdio` with `deployment.cmd: npx` / package args over runtime `npm install` in the pod or a wrapper image unless you truly need custom filesystem contents.
- KMCP packaging: if a kagent/RemoteMCPServer chat flow must create MCP sessions through AgentGateway, avoid KMCP `transportType: stdio` for npm-launched packages such as `@modelcontextprotocol/server-everything`; per-session `npx` startup can miss the ~5s session bootstrap window. Prefer a long-lived HTTP/server mode (`transportType: http` / `streamableHttp`) or a prebuilt image.
- Optional samples: do not include sample MCP workloads that depend on locally built or placeholder images in the default `platform-applications` path; gate them behind an explicit opt-in overlay or post-bootstrap workflow so a cold cluster can reach `Ready=True` without extra image import steps.
- Local operator access: verify Makefile port-forward targets against live `kubectl get svc -A` output; current local access uses `kagent-kagent-ui`, `kagent-kagent-controller`, and the gateway-facing `agentgateway-proxy` Service, not the older short service names.
- AgentGateway access: for north-south/operator access prefer the gateway-facing `Service/agentgateway-proxy` when it exists; the internal `agentgateway-system-agentgateway` service is not the canonical external entrypoint.
- K3s + Istio CNI: set HelmRelease values `cniConfDir=/var/lib/rancher/k3s/agent/etc/cni/net.d`, `cniBinDir=/var/lib/rancher/k3s/data/cni`, and `ambient.enabled=true` when using `ztunnel`; otherwise `istio-cni` may look healthy while `ztunnel` never gets `/var/run/ztunnel/ztunnel.sock`.
- TEI restart behavior: for local CPU embeddings, keep TEI on an ONNX-backed model, mount `/data` on an `emptyDir` cache, opt the pod out of ambient, and allow a longer Helm timeout; cold restarts otherwise flap on Hugging Face download retries and block `platform-infrastructure`.
- TEI CPU defaults: use an ONNX-backed embedding model for `EMBEDDING_MODEL` and generated `tei-values`; models without `model.onnx` keep `tei-embeddings` in rollout even after Flux and Helm are otherwise healthy.
- KServe chart behavior: upstream `ghcr.io/kserve/charts/kserve:v0.16.0` always renders the built-in `ClusterServingRuntime` objects and only toggles `spec.disabled`; in this repo, strip those objects with Helm post-renderer delete patches or stage them separately, otherwise first install can deadlock on the webhook before `kserve-controller-manager` has endpoints.
- Generated metadata like `clusters/<topology>-<env>/topology-values.yaml` is for operators only and must not be listed under Kustomize `resources`.
- Terraform/OpenTofu local_file: when generating tracked repo manifests such as `clusters/**` or `values/**`, set explicit `directory_permission="0755"` and `file_permission="0644"`; provider defaults can flip YAML files to executable and create noisy diffs.
